\section*{Resumen}

Esta práctica tiene tres partes con distinto grado de individualidad según la guía. La Parte 1A (segmentación con YOLOv11, admite pareja) la realizamos en equipo: entrenamos YOLOv11n-seg por transferencia de aprendizaje sobre un dataset propio de tres clases (señal de pare, semáforo, cono de tránsito), y medimos una aceleración de 6.5x de GPU frente a CPU (99.0 FPS vs 15.2 FPS).

Las Partes 1B y 1C son \textbf{individuales}: cada integrante las ejecutó en su propio equipo (MAC distinta) e implementó soluciones diferenciadas. En la Parte 1B ambos comparamos YOLOv12n y YOLOv26n para detección y usamos \textbf{RealPLKSR} (arXiv:2404.11848, abril 2024) para super resolución — cumpliendo el requisito de antigüedad menor a 2 años — pero con pesos y configuraciones distintos. En la Parte 1C cada uno implementó el pipeline de preprocesamiento en OpenCV CUDA C++ con kernel morfológico diferente (MORPH\_ELLIPSE vs MORPH\_CROSS) y distintas resoluciones de benchmark.

\section{Introducción}
% =============================================================================

Esta práctica reúne tres tareas de visión por computador aceleradas por GPU: segmentación de objetos por transferencia de aprendizaje, comparación de redes de detección y super resolución en vídeo, y un pipeline clásico de preprocesamiento en C++ con CUDA. El hilo conductor es el mismo en las tres partes: medir qué tan rápido corre una tarea en GPU frente a CPU, y entender \emph{por qué}.

La guía permite que la Parte 1A se resuelva en pareja, pero exige que las Partes 1B y 1C sean estrictamente individuales, con evidencia de un computador propio (MAC única). Por eso este informe distingue claramente qué sección corresponde a cada estudiante.

def benchmark_yolo(model_name, model_path, video_path, device, n_frames=100):
    model = YOLO(model_path)
    cap = cv2.VideoCapture(video_path)
    times = []
    while len(times) < n_frames:
        ret, frame = cap.read()
        if not ret: cap.set(cv2.CAP_PROP_POS_FRAMES, 0); continue
        t0 = time.perf_counter()
        model(frame, device=device, verbose=False, conf=0.4)
        times.append(time.perf_counter() - t0)

    return {'fps': 1.0/np.mean(times), 'avg_ms': np.mean(times)*1000}

\section{Parte 1C: Pipeline OpenCV CUDA C++ — CPU vs GPU-only (individual)}
% =============================================================================

Cada integrante implementó el pipeline de preprocesamiento de forma independiente, con diferencias en el kernel morfológico, el orden de operaciones y las resoluciones de benchmark. En ambos casos los filtros CUDA se crean \textbf{una sola vez} fuera del ciclo de medición (patrón \texttt{GpuPipeline}), y el pipeline GPU-only realiza un único \texttt{upload()} y un único \texttt{download()} por frame.

// Pipeline CPU: cvtColor -> GaussianBlur -> equalizeHist -> erode/dilate -> Canny
cv::Mat kernel = cv::getStructuringElement(cv::MORPH_ELLIPSE, cv::Size(3, 3));

// Pipeline GPU-only (filtros creados una vez en constructor):
struct GpuPipeline {
    explicit GpuPipeline(int type) {
        cv::Mat k = cv::getStructuringElement(cv::MORPH_ELLIPSE, cv::Size(3, 3));
        gauss    = cv::cuda::createGaussianFilter(type, type, cv::Size(5,5), 1.5);
        erode_f  = cv::cuda::createMorphologyFilter(cv::MORPH_ERODE,  type, k);
        dilate_f = cv::cuda::createMorphologyFilter(cv::MORPH_DILATE, type, k);
        canny    = cv::cuda::createCannyEdgeDetector(50, 150);
    }
};
\end{lstlisting}

\begin{jordybox}
\textbf{Equipo:} NVIDIA GeForce RTX 3080 Laptop GPU — MAC \texttt{d8:bb:c1:b4:24:3a}\\
\textbf{Kernel morfológico:} \texttt{MORPH\_CROSS}\\

\begin{lstlisting}
struct GpuPipeline {
    explicit GpuPipeline() {
        cv::Mat k = cv::getStructuringElement(cv::MORPH_CROSS, cv::Size(3,3));
        gauss    = cv::cuda::createGaussianFilter(CV_8UC1, CV_8UC1,
                                                  cv::Size(5,5), 1.5);
        erode_f  = cv::cuda::createMorphologyFilter(cv::MORPH_ERODE,  CV_8UC1, k);
        dilate_f = cv::cuda::createMorphologyFilter(cv::MORPH_DILATE, CV_8UC1, k);
        canny    = cv::cuda::createCannyEdgeDetector(50, 150);
    }
    cv::Mat process(const cv::Mat& frame, double& ms_out) {
        cv::cuda::GpuMat d_frame, d_gray, d_blur, d_eq, d_morph, d_edges;
        d_frame.upload(frame);                       // unico upload
        cv::cuda::cvtColor(d_frame, d_gray, cv::COLOR_BGR2GRAY);
        gauss->apply(d_gray, d_blur);
        cv::cuda::equalizeHist(d_blur, d_eq);
        erode_f->apply(d_eq, d_morph);
        dilate_f->apply(d_morph, d_morph);
        canny->detect(d_morph, d_edges);
        cv::cuda::Stream::Null().waitForCompletion();
        ms_out = elapsed_ms(t0);
        cv::Mat result; d_edges.download(result);    // unico download
        return result;
    }
};
\end{lstlisting}

\subsection{Reflexión: GPU-only vs pipeline alternado}

Un pipeline que alterna CPU y GPU en cada operación paga la transferencia PCIe en cada paso; con 5 operaciones, eso son 10 transferencias por frame frente a 2 en el GPU-only. La ventaja relativa de la GPU crece con la resolución (mayor volumen de datos) y con la cantidad de operaciones encadenadas.

\section{Conclusiones}
% =============================================================================

\begin{enumerate}

\item \textbf{La individualidad exige diferencias reales de código, no solo de resultado.} Los dos pipelines de Parte 1C usan kernels distintos (MORPH\_ELLIPSE vs MORPH\_CROSS), órdenes de operaciones distintos y resoluciones de benchmark distintas, lo que produce resultados comparables pero no idénticos.

\item \textbf{Elegir una red por su novedad no es un detalle burocrático.} Real-ESRGAN (2021) cumple la función pero no el requisito de antigüedad. RealPLKSR (2024) resultó ser un reemplazo directo compatible con \texttt{spandrel}, sin cambiar el resto del pipeline.

\item \textbf{Un benchmark ``GPU más lenta que CPU'' es motivo de sospecha.} Crear los filtros CUDA dentro del ciclo de medición hace que la GPU mida más lenta por el overhead de inicialización. El patrón correcto es crear los filtros una sola vez en el constructor y reutilizarlos.

\item \textbf{La aceleración GPU depende del equipo y de la tarea.} Michael (RTX 4060) obtuvo mayor aceleración en detección (8.1x--8.7x) y SR (23.7x) que Jordy (RTX 3080, 4.3x--4.4x detección, 7.4x--9.6x SR). La SR es la tarea que más se beneficia de la GPU en ambos casos por su mayor densidad de cómputo por píxel.

\item \textbf{Verificar el consumo de VRAM es tan importante como la velocidad.} RealPLKSR usó menos del 2\% de la VRAM disponible en ambos equipos, confirmando que el modelo puede escalar sin riesgo de \texttt{out of memory}.

\end{enumerate}
